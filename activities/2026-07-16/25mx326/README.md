# Agile for a New Software Engineer

## What is Agile?

Agile is simply a way software teams build products in **small, working pieces** instead of building everything at once.

Instead of spending months developing the entire product, teams:

- Build one feature
- Test it
- Get feedback
- Improve it
- Repeat

The goal is to deliver value continuously.

---

# How Work Actually Happens

The work is split into small features.

Example:

- User Login
- Product Search
- Add to Cart
- Payment
- Order Tracking

Each feature is completed in a short time period called a **Sprint** (usually 2 weeks).

# Complete Agile Software Development Lifecycle

## 1. Requirement Received

The client or business team requests a new feature.

Example:

> Customers should be able to save products to a wishlist.

---

## 2. Requirement Analysis

The Product Owner discusses:

- Business value
- Scope
- Priority

Output:

- User Story
- Acceptance Criteria

---

## 3. Product Backlog

All features are stored in a prioritized list called the **Product Backlog**.

Example:

- Login
- Wishlist
- Cart
- Payment

---

## 4. Sprint Planning

The team decides:

- Which backlog items to build
- Estimated effort
- Sprint goal

Output:

- Sprint Backlog

---

## 5. Development

Developers:

- Pick tasks
- Create a feature branch
- Write code
- Commit changes
- Push to Git
- Create a Pull Request

---

## 6. Code Review

Senior developers review:

- Code quality
- Logic
- Best practices

If approved:

- Merge into the main branch

---

## 7. Testing

QA verifies:

- Functionality
- Edge cases
- Bugs

If issues are found:

- Bug ticket created
- Developer fixes it
- QA retests

---

## 8. Deployment

After approval, the feature is deployed to:

- Development
- Staging
- Production

---

## 9. Feedback

Users provide feedback.

New improvements become new backlog items.

The cycle repeats.

# Scrum

Scrum is the most common Agile framework.

It organizes work into short development cycles called **Sprints**.

Usually:

- Sprint Duration: 2 weeks
- Goal: Deliver working software

---

# Scrum Roles

## Product Owner (PO)

Responsible for:

- Defining requirements
- Prioritizing backlog
- Accepting completed work

---

## Scrum Master

Responsible for:

- Running Scrum meetings
- Removing blockers
- Ensuring Scrum practices are followed

---

## Developers

Responsible for:

- Designing
- Coding
- Reviewing
- Fixing bugs

---

## QA/Test Engineer

Responsible for:

- Testing features
- Reporting bugs
- Verifying fixes

---

# Scrum Meetings

## Sprint Planning

Purpose:

- Decide what to build in this sprint.

Output:

- Sprint Backlog

---

## Daily Stand-up

15-minute daily meeting.

Each member answers:

- What did I complete?
- What am I working on today?
- Any blockers?

---

## Sprint Review

Purpose:

- Demonstrate completed features to stakeholders.
- Gather feedback.

---

## Sprint Retrospective

Purpose:

- Discuss what went well.
- Identify problems.
- Plan improvements for the next sprint.

---

# Important Documents

## Product Backlog

Complete list of all project requirements.

---

## Sprint Backlog

Tasks selected for the current sprint.

---

## User Story

Describes a feature from the user's perspective.

Example:

> As a customer, I want to save products to my wishlist so that I can buy them later.

---

## Acceptance Criteria

Conditions that must be met for a feature to be considered complete.

Example:

- User can add a product to the wishlist.
- Wishlist persists after login.
- Duplicate products are not added.

---
